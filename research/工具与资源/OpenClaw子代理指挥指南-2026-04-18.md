## sessions_spawn vs claude_session_start

### 何时用 sessions_spawn（runtime="subagent"）

`sessions_spawn` 是 **OpenClaw 原生的子代理工具**，适合：

| 场景 | 说明 |
|------|------|
| **从当前 Agent 派发并行任务** | 主 agent → 多个子 agent 并行执行研究/工具调用，不阻塞主会话 |
| **后台任务 + 自动汇报** | 子代理完成后自动 announce 结果回 requester 聊天频道 |
| **短生命周期任务** | 一次性执行，不需要持续监听用户后续消息 |
| **嵌套编排（orchestrator 模式）** | 主 → 编排者子代理 → 工作子子代理，结果沿链路上报 |

核心特点：
- 非阻塞，立即返回 `{ status: "accepted", runId, childSessionKey }`
- 完成后 push 通知 requester，无需轮询
- 默认会话隔离，子代理不继承主代理的工具权限
- 可配置 `runTimeoutSeconds` 超时控制
- 可配置 `cleanup: "delete"|"keep"` 清理策略

### 何时用 claude_session_start

`claude_session_start` 启动一个**持久化的 Claude Code 会话**，适合：

| 场景 | 说明 |
|------|------|
| **需要交互式 Coding** | 在持久会话中持续对话、编辑文件、运行命令 |
| **需要 MCP 工具** | Claude Code 原生支持 MCP 服务器 |
| **需要长时间保留会话状态** | 用户可能随时回来继续同一对话 |
| **需要自定义 Engine** | Codex / Gemini CLI / Cursor 等外部 CLI |

> ⚠️ `claude_session_start` 是**外部进程管理工具**，不是子代理工具。它启动的是独立 CLI 进程，主 agent 通过 `claude_session_send` 与其通信。

### mode="run" vs mode="session" 的区别

```
sessions_spawn 参数对照：

mode="run"（默认）
├── 一次性任务，执行完即结束
├── 结果通过 announce 自动上报
└── 适合：研究任务、批量工具调用

mode="session"（需 thread: true）
├── 持久会话，绑定频道线程
├── 后续同线程消息继续路由到该子代理
├── 支持 /focus /unfocus /session idle /session max-age
└── 适合：需要多轮交互的任务（如实时调试）

注意：mode: "session" 必须在 thread: true 时才生效
      若 thread: true 但 mode 省略，默认变为 "session"
```

## 任务分解最佳实践

### 核心原则

```
复杂任务 → 多个简单子任务 → 每个子代理只做一件事
```

### 实践指南

#### 1. 任务粒度控制

**✅ 好：单一职责子任务**
```
子代理A：调研 Topic A，输出结构化摘要
子代理B：调研 Topic B，输出结构化摘要
子代理C：汇总 A+B，输出综合报告
```

**❌ 差：一个大任务塞给一个子代理**
```
子代理：调研全部内容并写报告
→ 上下文窗口压力大，容易超时，结果质量不稳定
```

#### 2. 并行 vs 串行

```
并行：多个独立研究任务 → sessions_spawn x N（并发）
     例：同时调研3个竞品

串行：依赖链（后续任务依赖前序结果）→ 链式调用
     例：先搜索 → 再分析 → 再写报告
```

#### 3. 上下文传递技巧

**方法 A：在 task 描述中传递所有必要上下文**

```javascript
sessions_spawn({
  task: `
    背景：用户需要对比 A、B、C 三个方案
    研究目标：分析各方案优缺点，给出推荐
    格式要求：使用以下 Markdown 结构 [...]
    约束：只使用可信来源，标注不确定性
  `
})
```

**方法 B：通过 sessions_history 获取主会话历史**

```javascript
// 在子代理中主动拉取主会话上下文
sessions_history({ name: "agent:moling:main", limit: 10 })
```

**方法 C：label 标签追踪**

```javascript
sessions_spawn({ task: "...", label: "竞品A分析" })
sessions_spawn({ task: "...", label: "竞品B分析" })
sessions_spawn({ task: "...", label: "竞品C分析" })
// announce 回来时通过 label 识别来源
```

#### 4. 嵌套深度建议

| 深度 | Session Key | 角色 | 可派生子代理 |
|------|-------------|------|-------------|
| 0 | `agent:<id>:main` | 主代理 | ✅ |
| 1 | `agent:<id>:subagent:<uuid>` | 子代理/编排者 | ✅ 仅当 maxSpawnDepth ≥ 2 |
| 2 | `agent:<id>:subagent:<uuid>:subagent:<uuid>` | 工作子代理 | ❌ |

**推荐配置**（大多数场景）：
```json
{
  "agents": {
    "defaults": {
      "subagents": {
        "maxSpawnDepth": 2,
        "maxChildrenPerAgent": 5,
        "maxConcurrent": 8
      }
    }
  }
}
```

## 结果追踪

### 查看子代理结果

#### 方式 1：自动 announce（推荐）

子代理完成后自动向 requester 频道推送结果，无需主动查询。

#### 方式 2：sessions_history

```javascript
sessions_history({
  name: "agent:moling:subagent:abc123",  // childSessionKey
  limit: 20
})
// 返回经过安全过滤的会话历史：
// - 去除 thinking tags
// - 去除 tool-call XML scaffold
// - 去除凭证/密钥
// - 大消息可截断
```

#### 方式 3：直接查看 transcript 文件

```bash
# announce 返回的 payload 包含 transcript 路径
/Subagents info <runId>
# 显示：transcript path, session key, timestamps
```

### 给子代理发消息

```javascript
// 在主代理中发消息给运行中的子代理
sessions_send({
  name: "agent:moling:subagent:abc123",
  message: "补充调研一下 X 方向的最新进展"
})
```

> 也可通过 `/subagents send <id|#> <message>` 命令行操作

### 超时 / 失败处理

#### 超时配置

```javascript
sessions_spawn({
  task: "...",
  runTimeoutSeconds: 900  // 15分钟超时
})
```

#### 失败状态

announce payload 中的 `Status` 字段：
- `completed successfully` — 正常完成
- `failed` — 执行失败
- `timed out` — 超时
- `unknown` — 未知状态

#### 失败后的处理

```javascript
// 主代理在 announce 中判断
if (status === "failed" || status === "timed out") {
  // 重新派发 或 降级处理
  sessions_spawn({ task: "原始任务的简化版本", label: "降级重试" })
}
```

#### 级联停止

```
/stop → 主会话停止 → 所有 depth-1 子代理停止 → cascade 到 depth-2
/subagents kill <id> → 该子代理停止 → cascade 到其子代理
/subagents kill all → 全部停止 + cascade
```

#### 子代理数量限制

| 限制项 | 默认值 | 说明 |
|--------|--------|------|
| `maxConcurrent` | 8 | 全局并发上限 |
| `maxChildrenPerAgent` | 5 | 每个 agent 最多活跃子代理数 |
| `maxSpawnDepth` | 1 | 嵌套深度上限（设为2启用编排模式）|

## 常见错误（反模式）

### ❌ 反模式 1：轮询等待子代理完成

```javascript
// ❌ 错误：在循环中轮询
while (true) {
  const sessions = sessions_list()
  if (sessions.find(s => s.key === childKey)?.status === "done") break
  await new Promise(r => setTimeout(r, 5000))
}

// ✅ 正确：依赖 announce 推送，不主动等待
sessions_spawn({ task: "..." })
// → 子代理完成后自动 announce 结果回来
```

### ❌ 反模式 2：子代理任务过于宽泛

```javascript
// ❌ 错误：一个子代理做所有事
sessions_spawn({
  task: "调研 AI 编程工具生态，包括市场现状、主要竞品、技术趋势，给出投资建议"
})

// ✅ 正确：分解为多个专注子任务
sessions_spawn({ task: "调研 Copilot/Codeium/Cursor 市场占有率", label: "市场分析" })
sessions_spawn({ task: "调研各工具技术架构差异", label: "技术分析" })
sessions_spawn({ task: "调研最新融资和估值", label: "融资分析" })
// 然后主代理汇总
```

### ❌ 反模式 3：忘记设置 runTimeout

```javascript
// ❌ 错误：无超时，长时间运行任务可能导致资源泄漏
sessions_spawn({ task: "深度扫描代码库所有文件" })

// ✅ 正确：设置合理超时
sessions_spawn({
  task: "深度扫描代码库所有文件",
  runTimeoutSeconds: 300  // 5分钟超时
})
```

### ❌ 反模式 4：在子代理中再次调用 sessions_spawn（深度超限）

```javascript
// ❌ 错误：maxSpawnDepth=1 时，子代理无法再派生子代理
// 主代理（depth 0）派发 → 子代理（depth 1）再派发 → 被拒绝

// ✅ 正确：如果需要多级编排，配置 maxSpawnDepth: 2
// 然后 depth-1 编排者可派发 depth-2 工作子代理
```

### ❌ 反模式 5：忽略 cleanup 策略

```javascript
// ❌ 错误：大量子代理任务后不清理，产生会话积累
sessions_spawn({ task: "..." })  // 无 cleanup 参数，保留会话

// ✅ 正确：一次性任务用完即删
sessions_spawn({
  task: "...",
  cleanup: "delete"  // announce 后立即删除会话
})

// ✅ 正确：需要保留记录的分析任务
sessions_spawn({
  task: "...",
  cleanup: "keep"  // 保留会话，auto-archive 60分钟后自动清理
})
```

### ❌ 反模式 6：子代理中误用 session 工具

```javascript
// ❌ 错误：子代理默认没有 sessions_spawn / sessions_history 等工具
// 除非是 depth-1 编排者（maxSpawnDepth=2 时）

// depth-1 编排者能用的工具（maxSpawnDepth=2）：
// - sessions_spawn, subagents, sessions_list, sessions_history

// depth-2 工作子代理能用的工具：
// - 所有普通工具，无 session 工具
```

### ❌ 反模式 7：announce 结果泄漏敏感信息

```javascript
// sessions_history 自动过滤敏感信息，但 direct transcript 不会
// 如果需要完整历史，自行处理敏感字段脱敏
```

## 快速参考

### sessions_spawn 核心参数

| 参数 | 必填 | 默认值 | 说明 |
|------|------|--------|------|
| `task` | ✅ | — | 子代理任务描述 |
| `label` | ❌ | — | 标签，用于追踪 |
| `agentId` | ❌ | 当前agent | 目标agent |
| `model` | ❌ | 继承主代理 | 子代理模型 |
| `thinking` | ❌ | 继承主代理 | 推理深度 |
| `runTimeoutSeconds` | ❌ | 0（无超时）| 超时秒数 |
| `thread` | ❌ | false | 是否绑定频道线程 |
| `mode` | ❌ | run | run 或 session |
| `cleanup` | ❌ | keep | delete 或 keep |
| `sandbox` | ❌ | inherit | inherit 或 require |

### /subagents 命令速查

```
/subagents list          # 列出当前会话的子代理
/subagents info <id|#>   # 查看详细信息（transcript路径等）
/subagents log <id|#>    # 查看输出日志
/subagents send <id|#> <msg>   # 发消息
/subagents steer <id|#> <msg>  # 引导（替换任务描述）
/subagents kill <id|#|all>     # 停止（all含级联）
/subagents spawn <agentId> <task> [--model X] [--thinking level]
```
