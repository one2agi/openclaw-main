## ⚡ 一句话结论

OpenClaw 调校的核心是 **"95% 的能力来自配置框架本身，5% 来自模型"**（OnlyTerp 优化指南，2026年4月）。系统性地调校 SOUL/AGENTS/MEMORY/Skills 文件、控制 Context 大小、正确使用多 Agent 协作模式，是从"能用"到"好用"的三板斧。

## 元方法一：Context 大小管控（Context Engineering）

### 核心原理

每次对话注入的 bootstrap 文件（Soul/AGENTS/MEMORY 等）直接决定 Context 占用量。Context 越大 → 响应越慢 → 成本越高 → 质量越差。**控制 Context 体量是最立竿见影的优化手段。**

### 具体操作步骤

1. **设置 bootstrap 文件大小上限**（`openclaw.json`）：
   ```json5
   {
     agents: {
       defaults: {
         bootstrapMaxChars: 20000,      // 单文件上限，默认 20000
         bootstrapTotalMaxChars: 150000, // 总注入上限，默认 150000
       }
     }
   }
   ```

2. **强制截断警告**：
   ```json5
   { agents: { defaults: { bootstrapPromptTruncationWarning: "once" } } }
   ```

3. **定期精简 MEMORY.md**：生产级上限建议：
   - SOUL.md: < 1 KB（身份/语气）
   - AGENTS.md: < 2 KB（操作规则）
   - MEMORY.md: < 3 KB（持久事实）
   - 总计建议 < 5 KB（非 15 KB+ 的默认膨胀状态）

4. **使用 `/context list` 诊断**：查看每个注入文件的实际 Context 占用。

### 适用场景

几乎所有 Agent，尤其 Context 窗口较小的模型（如 16K token 模型）和慢响应场景。

### 风险/注意事项

- 截断可能导致 Agent 丢失关键身份/规则信息
- MEMORY.md 太小会导致 Agent 忘记重要偏好
- 建议配合 memory_search 工具从磁盘读取额外上下文

## 元方法二：Bootstrap 文件工程（Prompt Layering）

### 核心原理

OpenClaw 的 prompt 系统采用分层注入：**Bootstrap 层**（用户可控）→ **System Prompt 层**（框架生成）。用户通过正确填充 `SOUL.md`、`AGENTS.md`、`TOOLS.md`、`MEMORY.md`、`IDENTITY.md` 来影响 Agent 行为。分层清晰是保持 Agent 行为稳定的前提。

### 具体操作步骤

#### SOUL.md — 人格层（每次消息都注入）
**原则：短、精、有立场，不写 changelog 或安全政策**

```markdown
# SOUL.md — [Agent名]

## 核心价值观
[3-5条驱动行为的核心原则]

## 语气与边界
[说话风格、简洁度、幽默感]

## 底线
[绝对不做的事，1-3条]
```

**避坑**：
- ❌ 写"保持专业""提供积极体验"等废话规则
- ✅ 写具体行为指令："回复简洁，一句话说清的不说两句"

#### AGENTS.md — 操作规程层（每次消息都注入）
**原则：决策树 + 工具路由，不堆砌规则**

```markdown
# AGENTS.md — 操作手册

## 核心职责
## 工作流程
## 工具使用策略
## 协作规则
## 红线
```

#### TOOLS.md — 工具使用约定层
记录工具使用偏好、环境别名、设备名称等用户特定配置。

#### IDENTITY.md — 身份卡片
名字、Emoji、角色、上级。

#### HEARTBEAT.md — 心跳检查清单
Agent 在心跳周期内需要定期检查的事项列表。

### 适用场景

所有 Agent，多 Agent 协作中不同 Agent 需要明确角色分工时尤为重要。

### 风险/注意事项

- AGENTS.md 和 SOUL.md 职责重叠是常见错误
- SOUL/AGENTS/MEMORY 合计 < 5KB 为生产级目标

## 元方法三：模型分层配置（Model Tiering）

### 核心原理

不是所有任务都需要最强模型。用正确的模型做正确的事，可以将 Token 消耗降低 60%（OnlyTerp 基准测试）。Subagent 默认继承父 Agent 模型，但可以显式配置更便宜的模型。

### 具体操作步骤

#### 基础模型配置
```json5
{
  agents: {
    defaults: {
      model: "anthropic/claude-sonnet-4-6",  // 主 Agent
    }
  }
}
```

#### Subagent 模型降级（节省 60%+ 成本）
```json5
{
  agents: {
    defaults: {
      subagents: {
        model: "openai/gpt-4.1-mini",
      }
    }
  }
}
```

#### 渠道级模型覆盖
```json5
{
  channels: {
    modelByChannel: {
      telegram: { "-1001234567890": "openai/gpt-4.1-mini" },
    }
  }
}
```

#### 模型引用规范
- 格式：`provider/model`（如 `anthropic/claude-sonnet-4-6`）
- 模型 ID 本身含 `/` 时（如 OpenRouter），必须带 provider 前缀

#### Compaction 模型独立配置
```json5
{
  agents: {
    defaults: {
      compaction: {
        model: "ollama/llama3.1:8b"  // 摘要用本地小模型
      }
    }
  }
}
```

### 适用场景

- 研究类 Subagent → 用便宜模型
- 代码生成 → 用编程优化模型
- 定时任务（cron）→ 用便宜模型

### 风险/注意事项

- Subagent 模型太弱可能导致任务失败
- 不同渠道的消息量差异大，可按需配置

## 元方法四：Memory 系统精细化管理

### 核心原理

OpenClaw 的记忆本质是**磁盘上的 Markdown 文件**，Agent 只"记住"显式写入磁盘的内容。记忆分为三层：Bootstrap 层（每次注入）+ 每日文件层（按需读取）+ 语义搜索层（memory_search）。

### 具体操作步骤

#### 记忆文件分层策略
| 文件 | 用途 | 大小建议 | 谁写 |
|------|------|----------|------|
| `SOUL.md` | 身份+语气 | < 1 KB | 人类 |
| `AGENTS.md` | 操作规则 | < 2 KB | 人类+Agent |
| `MEMORY.md` | 持久事实 | < 3 KB | Agent（promote） |
| `DREAMS.md` | 反思日记 | 滚动 N 条 | Agent dreaming |
| `memory/YYYY-MM-DD.md` | 每日笔记 | 滚动 | Agent+自动捕获 |
| `vault/` | 原始资料 | 无上限 | 自动+人类 |

#### 语义搜索配置（memory_search）
```json5
{
  agents: {
    defaults: {
      memorySearch: {
        provider: "openai",       // 或 bedrock/gemini/voyage/ollama
        model: "text-embedding-3-small",
      }
    }
  }
}
```

**本地优先**（<100ms 延迟 vs 云端 2-5s）：
```json5
{ agents: { defaults: { memorySearch: { provider: "ollama", model: "nomic-embed-text" } } } }
```

#### Dreaming 配置（后台记忆整合）
```json5
{
  plugins: {
    entries: {
      "memory-core": {
        config: {
          dreaming: {
            enabled: true,
            schedule: "0 3 * * *",
          }
        }
      }
    }
  }
}
```

#### 每日记忆回填（Grounded Backfill）
```bash
openclaw memory rem-backfill --path ./memory --stage-short-term
# 回滚：
openclaw memory rem-backfill --rollback-short-term
```

### 适用场景

- 需要 Agent 跨会话记住用户偏好 → MEMORY.md
- 大量每日研究内容 → memory/YYYY-MM-DD.md + memory_search
- 长期知识积累 → DREAMS.md + Dreaming 自动提升

### 风险/注意事项

- MEMORY.md 无上限增长是 Context 污染主因之一
- `memory/*.md` 日常文件默认**不注入** Context
- Compaction 前会自动触发 memory flush

## 元方法五：多 Agent 协作模式（Orchestration Patterns）

### 核心原理

OpenClaw 提供四种多 Agent 协作机制：Subagent（轻量并行）、Council（多视角决策）、Thread-bound Subagent（持续性对话）、ACP Runtime（外部 Agent 集成）。

### 具体操作步骤

#### 模式一：Subagent（轻量并行）

```json5
// 配置 subagent 默认模型
{ agents: { defaults: { subagents: { model: "openai/gpt-4.1-mini" } } } }
```

触发方式：
- `/subagents spawn <agentId> <task> --model <model>`
- 工具调用 `sessions_spawn`

关键特性：
- 在独立 session 运行
- 完成后**自动推送结果**到请求者频道（push-based，勿轮询）
- Subagent 默认不继承 session tools（安全隔离）

#### 模式二：Council（多 Agent 委员会）

```json5
council_start({
  task: "评估 X 方案的可行性，输出推荐",
  projectDir: "/path/to/project",
  agents: [
    { name: "规划师", emoji: "📋", persona: "擅长拆解问题，制定计划" },
    { name: "工程师", emoji: "🔧", persona: "擅长代码实现，评估技术风险" },
    { name: "评估师", emoji: "⚖️", persona: "擅长批判性分析，找出漏洞" },
  ],
  maxRounds: 15
})
```

工作流程：每个 Agent 在独立 git worktree 中运行 → 轮次制 + 共识投票 → `council_review` → `council_accept` 或 `council_reject`

#### 模式三：Thread-bound Subagent（持续性对话）

```json5
sessions_spawn({
  task: "深度研究 X 主题",
  thread: true,       // 绑定到频道线程
  mode: "session",    // 持续性会话
})
```

**渠道支持**：目前仅 Discord 支持线程绑定。

#### 模式四：Sessions Spawn with ACP（外部 Agent 集成）

```json5
sessions_spawn({ task: "...", runtime: "acp", agentId: "custom" })
```

### 适用场景对比

| 模式 | 适用场景 | 隔离性 | 持续性 |
|------|----------|--------|--------|
| Subagent | 并行研究、慢任务分解 | 高 | 一次性 |
| Council | 多视角决策、代码评审 | 最高（worktree） | 多轮 |
| Thread-bound | 长任务、多轮对话 | 中 | 持续 |
| ACP Runtime | 深度编码任务 | 外部 CLI | 外部 |

### 风险/注意事项

- Subagent 不适合频繁状态同步，用 Task Flow 代替
- Council 的 `council_reject` **不会删除 worktree**
- 线程绑定目前仅 Discord 支持

## 元方法六：工作流自动化（Automation Stack）

### 核心原理

OpenClaw 提供五种自动化机制：**Heartbeat（近似周期）→ Cron（精确定时）→ Hooks（事件驱动）→ Standing Orders（永久指令）→ Task Flow（多步骤编排）**。

### 具体操作步骤

#### 1. Cron（精确定时任务）
```bash
openclaw cron add --name "daily-report" --cron "0 9 * * *" --task "生成报告"
```

#### 2. Heartbeat（近似周期监控，30分钟默认）
```json5
{
  agents: {
    defaults: {
      heartbeat: {
        every: "30m",
        showOk: false,
        showAlerts: true,
      }
    }
  }
}
```

`HEARTBEAT.md` 示例：
```markdown
# HEARTBEAT.md
## 定期检查
- [ ] 检查是否有新邮件需要回复
- [ ] 查看是否有 cron 任务失败
```

#### 3. Hooks（事件驱动）
```json5
{ hooks: { dirs: ["~/.openclaw/hooks"] } }
```

Hook 类型：`/new`、`/reset`、`/stop`、`session:compaction`、`gateway:start`、`message:receive`、`tool:call`

#### 4. Standing Orders（永久操作指令）
写在 `AGENTS.md` 中的持久指令，如：每天 18:00 发送工作日志摘要。

#### 5. Task Flow（多步骤编排）
```json5
{ agents: { defaults: { taskflow: { enabled: true, mode: "managed" } } } }
```
支持 `openclaw tasks flow list|show|cancel` 审查。

### 适用场景对比

| 机制 | 精确度 | Session 上下文 | 典型用途 |
|------|--------|----------------|----------|
| Heartbeat | ~30min | 完整 | 收件箱、通知 |
| Cron | 精确 | 独立或共享 | 每日报告、定时提醒 |
| Hooks | 事件触发 | 事件相关 | 生命周期脚本 |
| Standing Orders | 永久 | 每次注入 | 合规检查、权限边界 |
| Task Flow | 多步骤 | 持久化 | 研究→汇总→交付 |

## 元方法七：工具策略（Tool Strategy）

### 核心原理

OpenClaw 的工具有两大来源：**内置工具**（read/exec/edit/write 等，始终可用）和 **Skills**（按需加载的专业能力）。合理配置 Skills 是增强 Agent 能力的核心手段。

### 具体操作步骤

#### Skills 加载路径（优先级从高到低）
1. Workspace: `<workspace>/skills`
2. Project agent skills: `<workspace>/.agents/skills`
3. Personal agent skills: `~/.agents/skills`
4. Managed/local: `~/.openclaw/skills`
5. Bundled
6. `skills.load.extraDirs`

#### Skills 安装
```bash
openclaw skills install <slug>
```

#### Skills 按渠道/话题分配
```json5
{
  channels: {
    telegram: {
      groups: {
        "-1001234567890": {
          topics: {
            "99": {
              skills: ["research", "web-search"],
              systemPrompt: "保持话题专注"
            }
          }
        }
      }
    }
  }
}
```

#### 工具使用策略原则
- 长等待任务用 **cron**，不用 exec 循环睡眠
- 长任务用 **sessions_spawn**，不轮询
- 大型任务拆分为 **Subagent**

### 风险/注意事项

- Skills 越多 → Context 占用增加
- Subagent 默认不继承父 Agent 的 Skills

## 元方法八：Compaction 与 Context 窗口管理

### 核心原理

OpenClaw 在会话接近 Context 上限时自动执行 **Compaction**：将早期对话汇总为摘要，保留近期尾部。Compaction 前自动执行 memory flush。

### 具体操作步骤

#### Compaction 模型独立配置
```json5
{
  agents: {
    defaults: {
      compaction: {
        model: "ollama/llama3.1:8b"
      }
    }
  }
}
```

#### 标识符保留策略
```json5
{
  agents: {
    defaults: {
      compaction: {
        identifierPolicy: "strict"  // 保留代码/ID
      }
    }
  }
}
```

### 风险/注意事项

- Compaction 依赖主模型能力，小模型可能产生质量较低的摘要
- Context 溢出错误会触发 auto-compaction + retry

## 元方法九：安全与访问控制

### 具体操作步骤

#### 渠道访问控制
```json5
{
  channels: {
    telegram: {
      dmPolicy: "allowlist",
      allowFrom: ["tg:123456789"],
      groups: {
        "-1001234567890": {
          allowFrom: ["@admin"],
          requireMention: true
        }
      }
    }
  }
}
```

#### Exec 审批配置
```json5
{
  tools: {
    exec: {
      ask: "on-miss",
      security: "allowlist",
      applyPatch: false
    }
  }
}
```

#### 沙箱隔离
```json5
{
  agents: {
    defaults: {
      sandbox: {
        enabled: true,
        workspaceRoot: "~/.openclaw/sandbox"
      }
    }
  }
}
```

### 风险/注意事项

- `allowFrom: ["*"]` + `dmPolicy: "open"` = 完全开放（仅测试用）
- `tools.exec.ask: "off"` = 完全跳过审批（高危）
- Safety guardrails 在 System Prompt 中是**建议性**的

## 元方法十：Bootstrap 文件迭代法（The Molty Method）

### 核心原理

官方推荐用 Agent 自我迭代来改进 SOUL.md：将性格改造 prompt 发送给 Agent，让它自己重写自己的灵魂文件。

### 具体操作步骤

发送以下 prompt 让 Agent 重写 `SOUL.md`：

```markdown
Read your `SOUL.md`. Now rewrite it with these changes:
1. You have opinions now. Strong ones. Stop hedging everything.
2. Delete every rule that sounds corporate.
3. Add: "Never open with Great question, I'd be happy to help, or Absolutely."
4. Brevity is mandatory.
5. Humor is allowed when it fits naturally.
6. You can call things out. Charm over cruelty, but don't sugarcoat.
7. Swearing is allowed when it lands.
8. End with: "Be the assistant you'd actually want to talk to at 2am. Not a corporate drone. Not a sycophant. Just... good."

Save the new `SOUL.md`.
```

### 风险/注意事项

- Molty prompt 可能让 Agent 变得过于随意，需在 AGENTS.md 中加边界
- 公共频道/客户场景使用需评估语气是否合适

## 基础配置 vs 高级优化速查表

### 基础配置（必须做）

| 配置项 | 作用 | 推荐值 |
|--------|------|--------|
| `agents.defaults.workspace` | Agent 工作目录 | `~/.openclaw/workspace` |
| `channels.*.allowFrom` | 渠道白名单 | 实际用户 ID |
| `agents.defaults.model` | 主模型 | 按需求选型 |
| Bootstrap 文件初始化 | Agent 身份与规则 | SOUL/AGENTS/IDENTITY |

### 进阶优化（推荐做）

| 优化项 | 效果 |
|--------|------|
| Bootstrap 文件 < 5KB | Context 减少 60%+，速度提升 |
| Subagent 降级模型 | Token 消耗降低 60%+ |
| 本地 embedding（Ollama） | 搜索延迟 < 100ms |
| Compaction 独立模型 | 长会话质量稳定 |
| Cron + Heartbeat 分离 | 精确任务不遗漏 |

### 高级优化（可选）

| 优化项 | 适用场景 |
|--------|----------|
| Council 多 Agent 决策 | 战略级分析、代码评审 |
| Task Flow 编排 | 复杂多步骤流程 |
| memory-wiki 插件 | 需要结构化知识库 |
| Dreaming 配置 | 长期记忆自动整合 |
