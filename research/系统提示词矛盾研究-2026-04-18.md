## ⚡ 一句话结论

当系统提示词出现矛盾时，LLM 会**静默地通过内部启发式"抹平"冲突**，不报错、不警告——这使得矛盾极难被 agent 自己发现，OpenClaw 的多文件模块化架构在减少单文件矛盾的同时，也在文件组合边界处引入了新型冲突。

## 一、矛盾类型分类

### 1. 优先级冲突（Priority Conflicts）

| 层级 | 来源 | 正常优先级 |
|------|------|-----------|
| 最高 | OpenClaw Runtime System Prompt | 系统级指令 |
| ↓ | AGENTS.md（运营规则） | 应用级规则 |
| ↓ | SOUL.md（人格定义） | 角色定义 |
| ↓ | TOOLS.md / SKILL.md | 工具指导 |
| ↓ | 用户指令（User Message） | 用户请求 |
| 最低 | 工具输出（Tool Output） | 外部信息 |

**理论背景**：OpenAI（2024）提出指令层级（Instruction Hierarchy），主张系统提示词 > 用户指令 > 工具输出。但 NeurIPS 2025 的研究表明，**模型实际上经常忽略这一优先级**，转而服从权威性、专业性、社会共识等社会性线索。这意味着同一个 prompt 中的矛盾指令，最终被哪个覆盖并不由层级决定，而由"谁听起来更权威"决定。

### 2. 指令矛盾（Direct Contradictions）

- **内部文件矛盾**：同一文件内两条指令逻辑相反
  - 例如："永远使用 TodoWrite" vs "禁止使用 TodoWrite"
  - 例如："主动执行任务" vs "必须等待用户确认后再执行"
- **跨文件矛盾**：AGENTS.md 和 SOUL.md 对同一行为给出不同指引
  - 例如：SOUL.md 定义"主动冒险"，AGENTS.md 定义"风险操作必须停手"

### 3. 角色定位冲突（Role Conflicts）

- **人格与规则冲突**：SOUL.md 定义的人格特质（热情、冒险）与 AGENTS.md 的操作约束（保守、规避风险）产生根本矛盾
- **多文件来源冲突**：bootstrap 阶段注入的多个文件，各自定义了略有不同的角色认知，导致 agent 在不同场景下行为不一致
- **OpenClaw 特有风险**：sub-agent 只注入 AGENTS.md 和 TOOLS.md（不注入 SOUL.md），但主 agent 全量注入——同一 agent 在主/子会话中可能呈现不同"人格"

## 二、LLM 如何处理矛盾——理论机制

### 2.1 静默抹平（Silent Smoothing）——最核心的发现

**Arbiter 论文（UBC + Georgia Tech, 2026年3月）** 揭示了一个根本性问题：

> *"当系统提示词包含矛盾指令时，执行中的 LLM 会静默地通过训练提供的启发式方法解决冲突。**不报错。不警告。** 矛盾持续存在，agent 的行为变成模型在给定调用中权重更高的那个指令的函数。"*

这意味着：
- **Agent 无法可靠地检测自身 prompt 的矛盾**——"抹平矛盾"正是使 LLM 有用的同一机制，这让它们作为自身审计者不可靠
- 矛盾的解决结果具有**随机性**（取决于模型、温度、上下文窗口状态）
- 用户观察到的是"行为不稳定"而非"矛盾报错"

### 2.2 指令层级（Instruction Hierarchy）理论与现实的差距

| 维度 | 理论（OpenAI 2024） | 现实（NeurIPS 2025） |
|------|-------------------|-------------------|
| 系统 > 用户 | 应遵从这个优先级 | 模型有时遵从权威性线索而非层级 |
| 矛盾时的行为 | 忽略低优先级指令 | 忽略系统指令，服从权威性用户 |
| 安全性 | 层级隔离可防止注入攻击 | 社会性线索可绕过层级 |

### 2.3 知识冲突中的推理失败

**ConflictQA 论文（2026年）** 研究了 RAG 系统中的跨源知识冲突，发现：
- 当面临矛盾证据时，LLM 对提示词的选择**更加敏感**
- LLM 倾向于完全依赖某一个证据源（KG 或文本），而非进行综合推理
- 这种行为直接映射到 OpenClaw 的多文件 bootstrap 场景：**当 AGENTS.md 和 SOUL.md 给出不同方向时，agent 会选择其中一个来源"全押"**

## 三、实际表现——OpenClaw 特有的矛盾场景

### 3.1 OpenClaw 的 Prompt 架构

OpenClaw 的 bootstrap 文件按以下顺序注入到 Project Context：

```
AGENTS.md → SOUL.md → TOOLS.md → IDENTITY.md → USER.md → HEARTBEAT.md → BOOTSTRAP.md → MEMORY.md
```

**关键特性**：
- 每个会话开始时**重新注入**（不是只在第一个会话）
- Sub-agent 只注入 AGENTS.md + TOOLS.md（文档明确说 SOUL.md 不注入）
- 单文件最大 12,000 字符，总注入上限 60,000 字符（bootstrapTotalMaxChars）

### 3.2 已知的 OpenClaw 矛盾/Bug

| Issue | 类型 | 来源 |
|-------|------|------|
| Sub-agent 不加载 SOUL.md、IDENTITY.md | 文档 vs 实践矛盾 | GitHub #24852 |
| Bootstrap 文件注入失败（回归 Bug） | 功能性 Bug | GitHub #53547 |
| 部分 API 不注入 SOUL.md/USER.md | 平台兼容矛盾 | GitHub #3775 |
| 文档说 SOUL.md 注入子代理，但实际不注入 | 文档错误矛盾 | GitHub #27038 |

### 3.3 社区观察到的矛盾表现

Reddit r/openclaw（2026年3月）热门帖的核心洞察：

> "你最终会进入一个状态——可能有矛盾的指令（system prompt 说'总是包含精确时间'，而你 AGENTS.md 里说'不要过度细节化'）。OpenClaw 使这一点更加明显，因为它实际上**同时尝试遵循所有指令**，矛盾很快就会暴露出来。"

**矛盾的实际表现**：
- Agent 在同一任务中行为不一致（有时主动，有时等待）
- 性能下降但无明显错误日志
- 模型行为随上下文窗口变化而变化（部分 bootstrap 被截断后矛盾消失/出现）
- 高上下文使用量（>60,000 字符注入）时，模型注意力分散，矛盾"被忽略"的可能性增加

## 四、Prompt 架构与失败类型（Arbiter 论文发现）

Arbiter 论文（2026年3月）分析了三大编码 Agent 系统提示词（Claude Code、Codex CLI、Gemini CLI），发现了以下架构相关的失败类型：

| 架构类型 | 特点 | 失败模式 |
|---------|------|---------|
| **Monolithic（单体式）** | 所有指令放在一个大 prompt | 增长级 Bug，出现在子系统边界 |
| **Flat（扁平式）** | 无结构，所有指令平铺 | 以一致性换取能力 |
| **Modular（模块化）** | 多个文件/片段组合 | 设计级 Bug，出现在组合接缝处 |

**OpenClaw 属于典型的 Modular 架构**——其设计（SOUL.md、AGENTS.md、TOOLS.md 分立）是模块化的，但：
- 文件之间没有明确定义的"接口契约"
- 矛盾可能出现在文件边界（一个文件说做 A，相邻文件说做 B）
- 组合接缝处缺乏显式优先级声明

## 五、应对策略

### 5.1 如何检测 Prompt 矛盾

**方法一：多模型交叉审查（Arbiter 方案）**
- 将 prompt 发送给多个不同的 LLM
- 给模糊指令："仔细阅读并指出你发现的任何有趣之处"
- 收集所有模型的发现，再发给下一轮
- 收敛标准：连续三个模型都拒绝继续发现新问题

**方法二：自省 Prompt**
- 发送指令让模型**列出**自己被要求做的所有事情
- 然后让模型**识别**这些要求之间的逻辑矛盾
- 注意：这依赖模型本身的能力，有局限性

**方法三：使用 /context list 诊断（OpenClaw 特有）**
- /context list：查看注入的文件及各自贡献量
- /context detail：查看详细截断和注入情况
- 可以发现哪些文件被截断了，以及总注入量是否接近上限

**方法四：定期"冷启动"测试**
- 清除会话，以最小上下文启动
- 观察 agent 在无历史干扰时的行为
- 如果行为与预期不符，可能是 prompt 内部的矛盾

### 5.2 如何修复矛盾

**优先级原则：**
1. AGENTS.md > SOUL.md —— 运营规则优先于人格定义
2. 后注入 > 先注入 —— OpenClaw 的注入顺序靠后的文件权重更高（理论上）
3. 显式声明优先级 —— 在文件顶部写明本文件的定位和优先级

**截断风险管理：**
- 将长内容拆分成多个文件，而非堆积在一个文件
- bootstrapTotalMaxChars 默认 60,000，实际建议控制在 40,000 以内
- MEMORY.md 应保持精简，核心记忆使用 memory_store 而非大量文本

### 5.3 最佳实践：避免矛盾的结构化方法

**原则一：单一职责分离**

| 文件 | 职责 | 禁止包含 |
|------|------|---------|
| SOUL.md | 人格、语调、价值观 | 操作规则 |
| AGENTS.md | 工作流程、决策规则 | 人格描述 |
| TOOLS.md | 工具使用说明 | 流程规则 |
| IDENTITY.md | 身份定义 | 任何规则 |

**原则二：跨文件禁止出现同类型指令**

- 如果 AGENTS.md 已经定义了工作流程，SOUL.md 就不应该再定义"何时主动行动"
- 如果 TOOLS.md 已经定义了 exec 使用规范，AGENTS.md 就不应该再定义"shell 命令的使用限制"

**原则三：定期审计**
- 每月用多模型审查工具检查 bootstrap 文件
- /context list 检查注入量变化
- 每次更新 AGENTS.md 后，验证 SOUL.md 是否仍然兼容

**原则四：Sub-agent 隔离**
- Sub-agent 只接收 AGENTS.md + TOOLS.md（这是 OpenClaw 的设计）
- 不要期望 sub-agent 拥有完整人格——为 sub-agent 任务提供足够的上下文在 prompt 中，而非依赖 SOUL.md

## 六、核心结论

1. **LLM 不会报错**：系统提示词矛盾时，LLM 通过内部机制静默解决，不产生任何错误信号
2. **行为不可预测**：矛盾的解决结果取决于模型权重、温度、上下文状态，同一矛盾在不同调用中可能产生不同行为
3. **Agent 无法自审**：Arbiter 论文明确指出——"解决矛盾的 Agent 不能是检测矛盾的 Agent"
4. **OpenClaw 模块化设计是双刃剑**：分离了关注点减少了单文件内的矛盾，但引入了文件边界处的组合矛盾风险
5. **最有效的防御是结构化**：明确文件职责、禁止同类型指令跨文件重复、在文件顶部声明优先级
