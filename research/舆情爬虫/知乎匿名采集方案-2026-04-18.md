# 知乎匿名采集方案

**研究日期：** 2026-04-18  
**研究者：** 墨灵 🔍

---

## 是否有匿名方案：（无）

知乎**没有**完全无需 Cookie、无需签名的公开匿名端点。所有有价值的 API 均强制要求：
- `x-zse-96` 动态签名（核心门槛）
- `d_c0` Cookie（用户身份标识，来自 `/udid` 接口首次访问时设置）
- `x-zse-93` 版本标识（固定值 `101_3_3.0`）

---

## 核心限制

| 限制项 | 说明 |
|--------|------|
| **x-zse-96 签名** | 动态签名 = MD5(固定盐 + 请求路径 + d_c0 + URL参数)，结果再经混淆层处理。**每次请求必须重新生成**，无法硬编码。 |
| **d_c0 Cookie** | 来自 `/udid` 接口的 `set-cookie`，需要先发一次请求获取，后续携带在 Cookie 中。 |
| **IP 频率限制** | 高频访问会触发验证码或封禁。 |
| **API 版本迭代** | 知乎历史上从 x-zse-93 → x-zse-96，签名算法会不定期更新，需要持续维护逆向代码。 |
| **无官方公开 API** | 知乎未提供任何面向第三方的公开 API，所有接口均为内部使用。 |

---

## 关键 API 端点

### 热榜相关

| 端点 | 说明 | 匿名可用 | 签名要求 |
|------|------|----------|----------|
| `GET /api/v4/featured_text_hot_adaptive` | 首页热榜推荐 | ❌ | 需要 x-zse-96 + d_c0 |
| `GET /api/v1/zhihu/web/fetch_hot_list` | 知乎热榜（第三方 TikHub 封装） | ⚠️ 需要 API Key | 付费 |
| `GET /www.zhihu.com/topic/{id}/hot` | 话题热榜页面 | ❌ HTML 渲染，需 JS | Selenium / 无头浏览器 |
| `GET /api/v4/search_v3` | 搜索接口 | ❌ | 需要完整 x-zse-96 + d_c0 |

### 签名参数详解

```
x-zse-93:     "101_3_3.0"                    ← 固定值，API 版本标识
x-zse-96:     "2.0_" + md5(拼接字符串)         ← 核心，动态生成
d_c0:         从 /udid 接口获取的 cookie 值    ← 首次访问知乎域名时由服务器 set-cookie
x-zst-81:     可选，部分接口需要                ← 次要参数
```

签名拼接字符串构造（简化）：
```
s = [x-zse-93, 请求路径(包含query), d_c0, URL参数(条件), x-zst-81(可选)]
    .filter(Boolean)
    .join("+")
signature = 自定义MD5混淆层(md5(s))
x-zse-96 = "2.0_" + signature
```

## x-zse-93 签名分析

### x-zse-93 vs x-zse-96

| 参数 | 性质 | 是否必须 |
|------|------|----------|
| `x-zse-93` | 固定值 `101_3_3.0`，表示 API 版本 | ✅ 必须，但简单 |
| `x-zse-96` | 动态签名：`2.0_` + MD5混淆结果 | ✅ **核心门槛** |
| `x-zst-81` | 可选，部分接口需要 | ⚠️ 视接口而定 |

### x-zse-93 是否必须？

**必须，但不是难点。** 它是固定值，可以直接硬编码。

**x-zse-96 是真正的门槛：**

1. **MD5 层**：对拼接字符串做标准 MD5 哈希
2. **混淆层**：MD5 结果再经过一层自定义字符变换（涉及 Base64 变种和位置交换）
3. **每次请求必须重新计算**：因为 URL 参数（时间戳、offset 等）会变化

### 逆向难度

- 需要从知乎的混淆 JS 中提取签名生成函数
- 算法会不定期更新（历史上 21 年~24 年已至少迭代两次）
- Node.js 环境补全（部分函数依赖浏览器环境）
- **技术成本：高；维护成本：持续**

## 可行性评估

### 场景一：完全匿名（无 Cookie + 无签名）

**⭐ 极低（1/5）**

- 没有任何公开端点可以绕过签名和 Cookie 访问知乎数据
- 热榜 API 无一例外需要 x-zse-96
- 尝试抓取 HTML 页面也会被反爬拦截或返回空内容

### 场景二：最小化 Cookie（仅 d_c0）+ 自行生成签名

**⭐⭐ 中低（2/5）**

- 技术上可行，但需要持续维护逆向代码
- d_c0 可从 `/udid` 接口获取（无需登录），但属于知乎的 Cookie 政策灰色地带
- 签名算法更新时爬虫会立即失效，需要快速响应
- IP 频率限制仍需处理

### 场景三：第三方付费 API

**⭐⭐⭐⭐ 高（4/5）**

- TikHub 等平台封装了知乎热榜 API，API Key 认证，无需关心签名
- 有免费额度可测试，适合 POC 阶段
- 商业使用需付费，但成本可控
- 稳定性依赖第三方服务

### 场景四：聚合热榜平台

**⭐⭐⭐⭐ 高（4/5）**

- 今日热榜（free-api.com）、uapis.cn 等提供知乎热榜聚合
- 无需注册或注册即用，有免费额度
- 数据可能不是实时热榜，而是定时快照
- 适合轻量级舆情监控

## 推荐方案

### 🥇 短期/轻量级：第三方聚合 API + 定时轮询

```
平台：今日热榜 / uapis.cn / free-api.com
成本：免费（或极低）
接入：HTTP GET 请求，API Key 认证
数据：知乎热榜 + 多平台聚合
```

**优点：** 零维护，立即可用  
**缺点：** 数据非实时，可能有延迟

### 🥈 中期/稳定自建：最小化 Cookie + 签名逆向

```
步骤：
1. 用无头浏览器或 HTTP 请求获取 d_c0（/udid 接口）
2. 逆向 x-zse-96 生成逻辑（Node.js 环境）
3. 定时任务带上签名 + d_c0 请求热榜 API
4. 结果存储到本地/数据库
```

**优点：** 自控数据源，可获取完整数据  
**缺点：** 签名算法更新需快速响应，维护成本

### 🥉 长期/高可用：第三方付费 API（TikHub 等）

```
平台：TikHub.io 知乎热榜 API
成本：付费（有免费额度）
接入：API Key，官方 SDK 或直接 HTTP
数据：完整热榜 + 话题热榜
```

**优点：** 稳定可靠，专业团队维护  
**缺点：** 依赖第三方，数据成本

---

## 风险提示

1. **法律风险**：知乎用户协议明确禁止未经授权的大规模数据抓取，商业使用需评估法律合规性
2. **技术风险**：签名算法随时可能更新，逆向方案需要持续投入
3. **稳定性风险**：自建爬虫可能被封 IP，建议配合代理池使用
4. **数据质量**：第三方 API 数据可能有延迟或缺失
