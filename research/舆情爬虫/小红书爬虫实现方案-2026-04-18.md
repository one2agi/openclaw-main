# 小红书爬虫实现方案

> 研究日期：2026-04-18 | 研究者：墨灵 🔍

---

## ⚡ 一句话结论

**2026年小红书爬虫已无"纯免签"方案，所有可行方案核心都是解决 `x-s` 系列签名（x-s、x-t、x-s-common、xsec_token）；最成熟落地方案是 `MediaCrawler`（基于 Playwright 保留登录态）+ `xhshow`（Python 纯算签名生成库）组合，零逆向门槛，稳定可用。**

---

## 📊 置信度：高

基于 GitHub 实际项目代码、多篇实测教程综合判断。`xhshow` 在 PyPI 有持续更新（2025年仍活跃），`MediaCrawler` GitHub 27k+ stars，有完整文档和活跃社区。

## 一、当前可行的实现方案

### 方案1：MediaCrawler（推荐 ✅ 最成熟）

- **开源库/工具**：[NanmiCoder/MediaCrawler](https://github.com/NanmiCoder/MediaCrawler)（~27.7k ⭐）
- **支持平台**：小红书、抖音、快手、B站、微博、贴吧、知乎
- **GitHub**：[https://github.com/NanmiCoder/MediaCrawler](https://github.com/NanmiCoder/MediaCrawler)

#### 实现思路

**核心：不搞 JS 逆向，而是用 Playwright 真实浏览器登录后，通过 CDP 协议在浏览器内执行 JS 表达式直接获取签名参数。**

1. 用 Playwright 打开 Chrome，扫码登录小红书，保存登录态（Cookie）
2. 通过 CDP 连接到已有 Chrome，复用登录态
3. 在浏览器内执行小段 JS 代码，从内存中提取 `x-s`、`x-t` 等签名
4. 用提取到的签名组装请求，直接调用小红书 API
5. 支持关键词搜索、指定笔记ID、用户主页、评论等全功能

#### 核心代码片段

```bash
# 安装 uv（Python 包管理器）
curl -LsSf https://astral.sh/uv/install.sh | sh

# 克隆项目
git clone https://github.com/NanmiCoder/MediaCrawler
cd MediaCrawler

# 安装依赖（自动处理 Node.js 依赖）
uv sync

# 开启 Chrome 远程调试
# Chrome 地址栏输入: chrome://inspect/#remote-debugging
# 勾选 "Allow remote debugging for this browser instance"

# 关键词搜索爬取
uv run main.py --platform xhs --lt qrcode --type search

# 指定笔记ID爬取
uv run main.py --platform xhs --lt qrcode --type detail

# 或启动 WebUI（端口8080）
uv run uvicorn api.main:app --port 8080 --reload
```

#### 优势
- **零逆向**：不需要逆向 JS 算法，无需理解加密逻辑
- **复用登录态**：CDP 模式直接用浏览器已有 Cookie，风控极低
- **多平台统一**：一份代码支持抖音、小红书等 7+ 平台
- **功能完整**：搜索、笔记详情、评论、用户主页、图片/视频下载全支持
- **活跃维护**：Pro 版本已发布，企业级架构

#### 劣势
- 需要 Chrome 浏览器环境（部分服务器无法部署）
- 需要 Node.js（16+）
- 依赖浏览器运行，资源消耗较高

### 方案2：xhshow + xhs 库组合（✅ 纯算法签名）

- **开源库1**：[Cloxl/xhshow](https://github.com/Cloxl/xhshow)（PyPI: `pip install xhshow`）
- **开源库2**：[ReaJason/xhs](https://github.com/ReaJason/xhs)（PyPI: `pip install xhs`）
- **核心能力**：Python 纯算逆向生成 `x-s`、`x-s-common`、`x-t`、`x-b3-traceid` 等签名

#### 实现思路

**核心：不需要浏览器，在 Python 中纯算法生成所有签名参数。**

`xhshow` 将原本需要浏览器执行 JS 才能得到的签名，全部用 Python 算法还原：
- `x-s` 签名（请求级签名）
- `x-s-common` 签名（通用签名）
- `x-t` 时间戳
- `x-b3-traceid` / `x-xray-traceid`

#### 核心代码片段（xhshow）

```python
# pip install xhshow
from xhshow import Xhshow
import requests

client = Xhshow()

cookies = {
    "a1": "your_a1_value",
    "web_session": "your_web_session",
    "webId": "your_web_id"
}

# GET 请求签名
headers = client.sign_headers_get(
    uri="https://edith.xiaohongshu.com/api/sns/web/v1/user_posted",
    cookies=cookies,
    params={"num": "30", "cursor": "", "user_id": "123"}
)

base_headers = {
    "User-Agent": "Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36",
    "Content-Type": "application/json"
}
base_headers.update(headers)

response = requests.get(
    "https://edith.xiaohongshu.com/api/sns/web/v1/user_posted",
    params={"num": "30", "cursor": "", "user_id": "123"},
    headers=base_headers,
    cookies=cookies
)
print(response.json())
```

#### 核心代码片段（xhs 库）

```python
# pip install xhs
from xhs import XhsClient

# 初始化（需要 cookie）
client = XhsClient(cookie="你的cookie")

# 搜索笔记
results = client.search_note(
    keyword="美食探店",
    page=1,
    page_size=20
)

# 获取笔记详情（需要 xsec_token）
note = client.get_note_by_id(note_id="笔记ID", xsec_token="xsec_token值")

# 获取用户笔记列表
user_notes = client.get_user_notes(user_id="用户ID", cursor="")
```

#### 优势
- 纯 Python，无浏览器依赖，适合服务器部署
- `xhs` 库 API 封装完善，上手简单
- `xhshow` 可与 `xhs` 配合使用，解决签名问题

#### 劣势
- 签名算法依赖 a1 cookie，长期稳定性取决于 cookie 有效期
- 需要维护有效的 cookie（登录态）
- `xsec_token` 仍需额外获取（见下方专节）

## 二、xsec_token 处理方案

### 1. 浏览器获取方案（最直接）

**方法**：通过浏览器访问小红书分享链接，在 DevTools Network 中提取

步骤：
1. Chrome 打开小红书分享链接
2. F12 → Network → 刷新页面
3. 搜索 `xsec`，找到包含 `xsec_token` 的响应字段
4. 提取备用

### 2. 程序化方案

- `xhs` 库支持传入 `xsec_token` 参数获取笔记详情
- **当前最新方案**：通过分享链接的 note_id 配合签名算法生成 `xsec_token`
- [Cialle/RedCrack](https://github.com/Cialle/RedCrack) 声称支持纯算生成

### 3. 无需 xsec_token 的端点/方法

**存在部分无需 xsec_token 的场景**：
- **搜索结果列表页**：部分搜索接口仅需 `x-s` 签名（xhshow 可生成）
- **用户主页笔记列表**：`/api/sns/web/v1/user_posted` 接口仅需 `x-s` 签名
- **首页信息流**：部分 Feed 接口仅需基础签名

**结论**：笔记详情页（`/api/sns/web/v1/feed`）通常强制需要 `xsec_token`，但笔记列表、搜索列表等可以用纯签名方案绕过。

### 4. xhshow 能完全替代浏览器获取吗？

**部分替代，但不完全**：

| 需求 | xhshow 能否解决 |
|------|----------------|
| x-s 签名生成 | ✅ 可以（纯算） |
| x-s-common 签名 | ✅ 可以（纯算） |
| x-t 时间戳 | ✅ 可以（纯算） |
| xsec_token 生成 | ⚠️ 部分方案/付费 |
| 登录态管理 | ❌ 需要 cookie |

`xhshow` 可以生成大部分请求签名，但 `xsec_token` 属于笔记级别的访问令牌，部分场景仍需从浏览器获取。

## 三、2025-2026 小红书反爬变化

### 主要变化

1. **`x-s` 算法持续升级**：2024年底至2025年，多次更新 x-s 签名算法，增加了混淆和反调试手段
2. **`xsec_token` 强制化**：笔记详情接口从可选变为强制需要 xsec_token，且有效期缩短
3. **分享链接短期失效**：永久分享链接变为带有效期的动态链接（2024年11月起）
4. **账号风控升级**：短时间内大量请求，账号会被临时限制
5. **评论区动态加载**：评论区不再一次性返回全部内容，需要模拟滚动触发加载
6. **登录态检测强化**：Selenium 等更易被检测，需要 stealth 脚本或 CDP 方案

### 应对策略

| 变化 | 应对方案 |
|------|---------|
| x-s 算法升级 | 持续关注 xhshow 更新，或切换 Playwright 方案 |
| xsec_token 强制化 | MediaCrawler 浏览器内提取，或购买第三方代采服务 |
| 分享链接失效 | 改用笔记 ID 直接请求（部分接口支持） |
| 账号风控 | 多账号轮换 + 请求间隔随机化 + 代理 IP |
| 登录态失效 | 定期刷新 cookie，或使用 MediaCrawler 自动维护 |

## 四、推荐实现步骤

### 步骤1：评估需求，选择方案

```
需求类型                          推荐方案
─────────────────────────────────────────────────
快速验证 / POC / 小规模采集        → MediaCrawler
服务器部署 / 自动化流水线          → xhs + xhshow 组合
大规模采集 / 企业级                → MediaCrawlerPro
专注下载（图片/视频）              → XHS-Downloader
需要笔记详情（含评论）             → MediaCrawler（优先）
```

### 步骤2：准备环境（MediaCrawler 为例）

```bash
# 1. 安装 uv
curl -LsSf https://astral.sh/uv/install.sh | sh

# 2. 安装 Chrome（版本 >= 144）
# 下载地址：https://www.google.com/chrome/

# 3. 开启 Chrome 远程调试
# Chrome 地址栏输入: chrome://inspect/#remote-debugging
# 勾选 "Allow remote debugging for this browser instance"

# 4. 克隆 MediaCrawler
git clone https://github.com/NanmiCoder/MediaCrawler
cd MediaCrawler
uv sync

# 5. 配置（编辑 config/base_config.py）
# 设置 ENABLE_GET_COMMENTS = True（如需评论）
# 设置爬取关键词、存储方式等
```

### 步骤3：获取小红书登录态

```bash
# 运行后弹出 Chrome 窗口，扫码登录
uv run main.py --platform xhs --lt qrcode --type search

# 程序会等待60秒，在弹出的 Chrome 窗口中扫码登录小红书
# 登录后程序自动继续
```

### 步骤4：执行爬取

```bash
# 关键词搜索爬取
uv run main.py --platform xhs --lt qrcode --type search

# 指定笔记详情爬取
uv run main.py --platform xhs --lt qrcode --type detail

# 或启动 WebUI 可视化操作
uv run uvicorn api.main:app --port 8080 --reload
# 访问 http://localhost:8080
```

### 步骤5：数据存储与后续处理

MediaCrawler 支持 CSV、JSON、JSONL、Excel、SQLite、MySQL 等多种存储方式，配置后自动导出。

## ⚠️ 局限性

- **cookie 有效期**：所有方案都需要维护有效的登录 cookie，定期刷新是必要的运维工作
- **签名算法时效性**：小红书频繁更新签名算法，依赖第三方库时需关注版本更新
- **大规模采集风险**：高频率请求仍可能触发风控，建议配合代理池
- **合规边界**：爬取公开数据用于个人学习研究是灰色地带，商业使用需自行评估法律风险
