# automation-design

设计任何自动化任务前的检查清单。

## 核心问题

**在动手之前，先回答这四个问题：**

### 1. 边界是否清晰？

| ✅ 清晰 | ❌ 模糊 |
|---------|--------|
| 固定输入 → 固定处理 → 固定输出 | "判断是否重复" |
| 读取文件A，写入文件B | "做有意义的事" |
| 每天 9:00 检查 X | "定期汇报" |

**如果无法用一句话说清楚"做什么、做到什么程度"，就不要继续。**

### 2. 执行者选对了吗？

| 执行者 | 最适合 | 最不适合 |
|--------|--------|---------|
| **Shell 脚本** | 文件操作、API 调用、条件判断 | 需要判断力的任务 |
| **Cron** | 固定周期、固定步骤 | 步骤不固定、需要 AI 推理 |
| **AI agent** | 写作、规划、创意生成 | 确定性执行、严格比对 |

**原则：只用 AI 在真正需要"判断力"的地方。**

### 3. 时间和失败可预估吗？

- 每次运行**最多需要多少时间**？
  - 如果无法估算 → 任务设计有问题
  - 留足冷启动余量（AI agent 首次加载 30-60s）
- **失败后怎么办**？
  - 明确日志输出
  - 部分成功如何处理
  - 需要人工介入的判断标准

### 4. 失败案例特征（反模式）

以下特征出现任意一个 → 重新设计：

```
❌ "让 AI 做严格比对/排重"
❌ "让 AI 做需要确定性输出的任务"
❌ "没有阶段性成果物，不知道跑了多少算成功"
❌ "步骤数量不固定，AI 自己决定做多少"
❌ "超时时间靠猜，没有估算依据"
```

## 设计检查清单

```
任务 X 适合自动化吗？
│
├─ 步骤是否清晰、固定？
│   └── 否 → 重新设计，先把流程写清楚
│
├─ 每次运行时间可预估吗？（含冷启动）
│   └── 否 → 设置上限或分阶段
│
├─ 失败后能自动恢复或明确记录吗？
│   └── 否 → 加日志和回退机制
│
├─ 真的需要 AI 吗？
│   ├── 是 → AI 只在判断节点介入，其余用脚本
│   └── 否 → 优先用纯脚本
│
└─ 失败的最坏结果是什么？
    ├── 破坏性操作 → 必须有人工确认步骤
    └── 可接受 → 自动化可以跑
```

## 好任务 vs 坏任务示例

### ✅ 好任务：MoltBook 运营

```
步骤1: 读取状态文件（确定性脚本）
步骤2: 运行互动脚本（确定性脚本）
步骤3: 更新状态文件（确定性脚本）
步骤4: 报告结果（AI 总结，200字）
---
超时: 5 分钟
失败: 写日志 → 下次继续
```

### ❌ 坏任务：每周记忆去重

```
让 AI 对多个文件做语义去重
---
问题: "语义去重"是模糊目标
问题: 每一步都是 AI 推理，不知道跑多久算完成
问题: 无法判断"去重成功"的标准
结果: 超时，不确定跑了多少
```

## 设计完成后

用一句话验证：
> "这个任务如果跑 10 分钟，我能否说清楚它做了什么、做到了什么程度？"

如果说不清楚 → 重新设计。

## 核心设计原则：两阶段设计

**AI 适合做判断，不适合做执行。**

两阶段设计模板：
```
阶段1（脚本，固定时间）：
  → 精确比对、数据处理、文件操作
  → 生成"疑似问题列表"
  → 写报告文件，不修改原始内容

阶段2（AI，可选）：
  → 读取报告
  → 只对"疑似问题"做判断
  → 用户确认后才操作
```

**为什么更好：**
- 阶段1固定时间，永不超时
- AI 只在最需要判断力的地方介入
- 用户控制权不丢失
- 失败有明确的"做到了哪里"
